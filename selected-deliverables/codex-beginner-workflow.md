# Codex Beginner Workflow

## Overview

This guide introduces a beginner-friendly workflow for using an AI coding or project assistant on small tasks. Codex is one example of this kind of tool, but the habits here can also apply to other AI assistants that help with planning, drafting, editing, explaining, or checking project work.

The main skill is not advanced software engineering. The main skill is clear planning, small tasks, careful review, and responsible use.

::: info Tool-agnostic framing
Learners do not need Codex access to use this workflow. The same habits can support many AI-assisted project tasks, including documentation, simple websites, diagrams, data cleanup, and planning artifacts.
:::

## Audience

| Audience | How this guide can help |
| --- | --- |
| Students new to AI-assisted project work | Learn how to describe a small goal and review AI-generated help. |
| Faculty and instructors | Introduce AI-assisted workflows without requiring advanced coding knowledge. |
| Workshop participants | Practice planning, reviewing, and iterating on a small artifact. |
| Teaching support staff | Prepare examples, checklists, and activity structures for beginner sessions. |

## Learning Goals

After using this guide, learners should be able to:

- Define a small project goal in plain language.
- Provide useful context, constraints, and a definition of done.
- Ask an AI assistant for a bounded first draft or explanation.
- Review outputs before accepting them.
- Test or check the result using simple evidence.
- Document what changed, what was checked, and what still needs review.

## What This Workflow Is For

This workflow is useful for small, reviewable tasks where the learner can inspect the result.

| Good fit | Example |
| --- | --- |
| Documentation | Turn notes into a clearer guide, checklist, or FAQ. |
| Small web or teaching artifact | Draft a simple explainer page or class demo. |
| Diagram or structure | Create a project map, module outline, or process diagram. |
| Data cleanup with public/sample data | Organize a small table, create a CSV, or draft a simple chart explanation. |
| Code explanation | Ask the assistant to explain generated code or a small file in plain English. |
| Revision support | Improve structure, naming, formatting, or readability. |

## What It Is Not For

This workflow should not be treated as a shortcut around review, policy, or learning.

| Not a good fit | Why to avoid or limit it |
| --- | --- |
| Private or sensitive data | Tool access and data handling may not be approved. |
| High-stakes decisions | Human expertise and institutional guidance should lead. |
| Full applications without support | Beginners may not be able to review security, data, or deployment risks. |
| Unchecked generated code | Working-looking code can still be wrong, insecure, or hard to maintain. |
| Course submissions without guidance | Learners must follow assignment rules, attribution expectations, and AI-use policy. |

## Beginner Workflow

| Step | What the learner does | Review habit |
| --- | --- | --- |
| 1. Choose a small task | Pick one output that can be read, opened, tested, or explained. | Keep the task low-stakes and limited. |
| 2. Explain the goal | State the audience, desired output, and why it matters. | Check whether the goal is specific enough. |
| 3. Add context and constraints | Provide relevant files, notes, tone, format, privacy limits, and time limits. | Do not include private data unless approved. |
| 4. Ask for a first draft or plan | Request a draft, outline, checklist, explanation, or small change. | Ask the assistant to state assumptions. |
| 5. Review the result | Read it, run it, compare it with the source, or test the main behavior. | Do not accept output just because it looks complete. |
| 6. Iterate simply | Ask for one or two concrete changes at a time. | Keep each revision easy to inspect. |
| 7. Document what happened | Note what the AI helped with, what changed, and what still needs checking. | Keep a short review trail. |

::: tip Beginner mindset
Treat the assistant like a helper that can draft and explain. The learner still owns the goal, judgment, and final decision.
:::

## Prompt Scaffold

Use this scaffold to ask for bounded help. Replace the bracketed text before using it.

```text
Goal:
I want to [make / revise / explain / organize / check] [specific output].

Audience:
This is for [students / workshop participants / instructor planning / my own learning].

Context:
Use [notes, files, topic, course context, or source material that is safe to share].

Constraints:
Keep [tone, length, format, privacy, policy, tools, time, accessibility] in mind.

Definition of done:
The result should [open, run, read clearly, match the source, follow the checklist, or include review notes].

Before you start:
State any assumptions and ask if something important is missing.
```

## Review Checklist

Before accepting AI-assisted project work, ask:

- [ ] Is the task small enough to review?
- [ ] Did I provide only safe, appropriate context?
- [ ] Does the output match the original goal?
- [ ] Did the assistant make assumptions I need to check?
- [ ] Did it change only what I intended?
- [ ] Can I open, run, read, or test the result?
- [ ] Are facts, citations, datasets, or claims verified?
- [ ] Is any generated code understandable enough to explain?
- [ ] Does the work follow classroom, workshop, or course guidance?
- [ ] Did I document what AI helped with and what I reviewed?

## Classroom or Workshop Use

This guide can support low-stakes activities that teach planning and review.

| Activity | How to use it |
| --- | --- |
| Project brief practice | Learners write a goal, audience, constraints, and definition of done. |
| Prompt scaffold activity | Learners turn a vague request into a clear project prompt. |
| Review exercise | Learners inspect an AI-generated outline, page, or code snippet using the checklist. |
| Iteration practice | Learners request one focused revision and explain why it improves the result. |
| Reflection note | Learners write what AI helped with, what they checked, and what they would not trust yet. |

Keep examples simple: a short guide, a one-page explainer, a public-data chart, a folder map, or a small non-sensitive demo.

## Next Steps

To adapt this workflow for teaching:

1. Choose a low-stakes task that learners can inspect.
2. Decide whether the activity is tool-agnostic or tied to a specific approved assistant.
3. Provide sample or public materials instead of private data.
4. Use the prompt scaffold to create a first draft or plan.
5. Review the result with the checklist.
6. Ask learners to document what they accepted, changed, rejected, or still need to verify.

This workflow should stay small, transparent, and reviewable. It is a beginner practice for responsible AI-assisted project work, not a requirement to use any particular tool.
