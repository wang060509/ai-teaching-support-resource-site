# Beginner AI-Assisted Project Workflow

## Overview

AI-assisted project workflow is a beginner-friendly way to plan, draft, review, and improve small digital projects with help from an AI assistant.

The informal term "vibe coding" is sometimes used for this kind of project-first work. In this guide, the focus is not on hype or instant app-building. The focus is on learning through small scope, clear goals, testing, screenshots, review, and iteration.

::: info Teaching focus
AI can help draft, debug, explain, and revise. The learner still needs to understand the goal, test the result, and decide what is ready to use.
:::

## Audience

| Audience | How this guide can help |
| --- | --- |
| Non-CS students | Start small digital projects without needing to write every line from scratch. |
| Faculty and instructors | Introduce AI-assisted making as a structured learning activity. |
| Workshop participants | Practice planning, reviewing, and improving a simple artifact. |
| Teaching support staff | Prepare low-stakes activities that connect AI use with reflection and review. |

## Learning Goals

After using this guide, learners should be able to:

- Define a small digital project with a clear audience and purpose.
- Plan before asking AI to build or revise.
- Use AI to draft, explain, debug, or improve a small artifact.
- Test the result and capture evidence such as screenshots or notes.
- Identify what still needs human review.
- Reflect on what AI helped with and what the learner changed.

## What "AI-Assisted Project Workflow" Means

AI-assisted project workflow means using AI as a support tool during a small project cycle.

| Phase | What the learner does | How AI can help |
| --- | --- | --- |
| Plan | Define the goal, audience, scope, and success criteria. | Ask clarifying questions or suggest a first outline. |
| Draft | Create a first version of the project, page, chart, guide, or demo. | Generate starter structure, sample text, rough code, or layout ideas. |
| Review | Inspect the output against the goal. | Explain what changed or identify possible issues. |
| Test | Open, run, read, compare, or try the main interaction. | Suggest checks or help interpret errors. |
| Iterate | Make one or two focused improvements at a time. | Revise a specific part while keeping the goal visible. |
| Document | Record what was tried, checked, and changed. | Help summarize the process in plain language. |

## What It Is Good For

This workflow is best for small, visible, low-stakes artifacts that learners can inspect.

| Good fit | Example |
| --- | --- |
| Concept explainer | A one-page HTML explanation of an AI literacy topic. |
| Simple class demo | A small interactive page for sorting examples or checking understanding. |
| Study support artifact | A glossary, checklist, practice prompt, or summary table. |
| Public/sample data activity | A simple chart or caption using non-sensitive data. |
| Draft website page | A small resource page that can be reviewed for clarity and accuracy. |
| Reflection artifact | A short note explaining prompts, revisions, screenshots, and checks. |

## What It Is Not Good For

AI-assisted project work should stay bounded for beginners.

| Not a good fit | Why to avoid or limit it |
| --- | --- |
| Large full-stack applications | Too many hidden risks for beginners to review. |
| Projects with private data | Privacy and policy approval may be required. |
| High-stakes tools | Errors may affect grades, access, safety, or decisions. |
| Unchecked publishing | Deployment can introduce privacy, copyright, security, and maintenance issues. |
| "Build anything instantly" expectations | Real learning still requires planning, testing, debugging, and explanation. |
| Code the learner cannot explain | Learners need enough understanding to review and maintain the artifact. |

## Beginner Workflow

| Step | Action | Beginner habit |
| --- | --- | --- |
| 1. Choose a small project | Pick one visible artifact, such as a page, checklist, chart, or demo. | Keep the first version simple. |
| 2. Define success | Name the audience, purpose, format, and definition of done. | Write this before prompting. |
| 3. Ask for a first draft | Request a small first version, outline, or plan. | Ask the AI to state assumptions. |
| 4. Open or inspect the result | Read the page, run the demo, check the chart, or review the file. | Do not judge by appearance alone. |
| 5. Test the main behavior | Try the main button, link, chart, example, or explanation. | Capture screenshots or notes. |
| 6. Revise one thing at a time | Ask for a specific improvement. | Keep changes easy to review. |
| 7. Explain what changed | Write a short reflection on prompts, revisions, tests, and remaining issues. | Learning comes from inspection, not just output. |

## Common Misunderstandings

| Misunderstanding | Better framing |
| --- | --- |
| "AI decides what to build." | The learner defines the goal, audience, and quality bar. |
| "If it looks polished, it works." | Visual polish does not prove accuracy, accessibility, or correct behavior. |
| "I do not need to understand the code." | Learners should understand enough to explain the main parts and risks. |
| "One long prompt should finish everything." | Small iterations are easier to inspect and improve. |
| "Debugging is failure." | Debugging is part of the learning workflow. |
| "Publishing is just one final click." | Publishing can raise privacy, copyright, policy, and maintenance questions. |

## Review and Safety Checklist

Before sharing or using an AI-assisted project, ask:

- [ ] Is the project scope small and clear?
- [ ] Did I use only public, sample, or approved materials?
- [ ] Can I explain the goal and audience?
- [ ] Can I explain the main parts of the output?
- [ ] Did I open, run, or inspect the result?
- [ ] Did I test the main interaction or claim?
- [ ] Did I capture screenshots, notes, or evidence of testing?
- [ ] Are facts, labels, data, and examples accurate?
- [ ] Is the result accessible and readable enough for the audience?
- [ ] Does the project follow course, workshop, or tool-use guidance?
- [ ] Do I know what still needs review before publishing or submitting?

## Classroom or Workshop Use

This guide can support low-stakes activities for non-CS learners.

| Activity | How to use it |
| --- | --- |
| Project idea sketch | Learners define a small artifact, audience, and success criteria. |
| Prompt-to-draft practice | Learners ask for a first version of a simple page, checklist, or explainer. |
| Screenshot review | Learners capture the result and mark what works, what is unclear, and what needs testing. |
| One-change iteration | Learners request one focused revision and compare before/after results. |
| Reflection note | Learners explain what AI helped with, what they checked, and what they still do not trust. |

Good first activities include a one-page concept explainer, a prompt checklist, a public-data chart, a study guide, or a simple interactive sorting activity.

## Next Steps

To adapt this workflow for teaching:

1. Choose a small artifact that fits the learning goal.
2. Provide sample content or public data.
3. Ask learners to write the goal, audience, constraints, and definition of done.
4. Use AI to create a first draft or plan.
5. Require screenshots, testing notes, or a short review checklist.
6. Ask learners to revise one part and explain the change.
7. Close with reflection on responsible AI use, verification, and human judgment.

This workflow should make AI-assisted project work more visible and reviewable. It should not be used to imply that beginners can skip understanding, testing, or responsible decision-making.
